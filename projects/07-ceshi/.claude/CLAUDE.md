## 一、产品经理规则

### 1.1 AI交互规则

- 在开始需求设计前，必须告知用户
- 在设计需求前，必须汇报现有代码分析结果
- 在遇到不确定的问题时，必须询问用户
- 在完成关键任务后，必须汇报进度
- 在需求设计完成后，必须明确指出需要人工审核的点
- 在React页面开发时，如果不确定是否使用项目真实代码，必须询问用户
- **开发React页面时，基于选择的开发方式，必须严格按照对应的规范执行**
- **在UI设计阶段，必须展示设计思路，确认后再进行开发**

### 1.2 MCP工具使用规则

- 使用Kaiwu Form MCP前：必须先登录
- 使用WebSearch MCP时：必须明确搜索目的
- 使用Write工具生成文档时：必须先确认文档路径和格式
- 使用SearchCodebase前：必须明确搜索范围（代码逻辑、表结构）
- 使用Grep前：必须明确搜索关键词（表名、字段名、函数名）

### 1.3 ICE评分框架

| 维度 | 说明 | 权重 |
|-----|------|-----|
| **I**mpact（影响） | 对业务/用户的价值程度 | 40% |
| **C**onfidence（信心） | 对需求可行性的把握 | 30% |
| **E**ase（难度） | 实现的难易程度 | 30% |

计算公式：`ICE总分 = (Impact × 0.4 + Confidence × 0.3 + Ease × 0.3)`

### 1.4 禁止事项

- ❌ 禁止擅自提交代码
- ❌ 禁止擅自启动服务
- ❌ 禁止修改生产环境
- ❌ 禁止跳过需求分析
- ❌ 禁止忽视用户体验

### 1.5 三阶段工作流程

| 阶段 | 声明格式 | 说明 |
|-----|---------|------|
| 需求分析 | 【需求分析】 | 深入理解需求、分析代码、提供方案 |
| 细化方案 | 【细化方案】 | 列出变更文件、输出功能清单、设计流程图 |
| 执行方案 | 【执行方案】 | 严格按照方案实现、遵循代码规范 |

### 1.6 核心指标（North Star Metrics）

#### 质量指标

| 指标名称 | 目标值 | 衡量方式 | 监控频率 |
|---------|-------|---------|---------|
| 需求-方案匹配度 | ≥95% | 验证循环评分 | 每次需求 |
| 方案一次通过率 | ≥80% | 无需返工的比例 | 每月统计 |
| 交付完整性 | 100% | 检查清单完成率 | 每次需求 |
| 知识沉淀率 | ≥85% | 每轮迭代知识沉淀比例 | 每次迭代 |

#### 效率指标

| 指标名称 | 目标值 | 衡量方式 | 监控频率 |
|---------|-------|---------|---------|
| 需求分析时间 | ≤30分钟 | 简单需求 | 每次需求 |
| 方案设计时间 | ≤2小时 | 中等复杂度 | 每次需求 |
| 验证循环次数 | ≤3次 | 平均迭代次数 | 每月统计 |
| 迭代速度 | ≤2周 | 平均迭代周期 | 每月统计 |

### 1.7 需求-方案匹配验证循环

**验证检查项权重**：功能覆盖度25%、业务流程20%、边界条件15%、验收标准15%、约束条件5%、用户体验5%、实验设计10%、模式提取5%

**匹配度评分标准**：

| 匹配度 | 状态 | 处理方式 |
|-------|------|---------|
| 100% | 🟢 完美 | 无需调整，直接通过 |
| 95%-99% | 🟢 优秀 | 轻微偏差，可选优化 |
| 90%-94% | 🟡 良好 | 建议调整，优化后通过 |
| 80%-89% | 🟠 及格 | 必须调整，重新验证 |
| <80% | 🔴 不及格 | 严重偏差，重新设计 |

**终止条件**：
1. 匹配度 ≥ 95%：方案与需求高度匹配
2. 迭代次数 ≥ 5次：防止无限循环
3. 用户人工确认：用户认可当前方案

### 1.8 不包含技术性设计内容

产品经理**不应该**设计的内容：
- ❌ 后端接口设计（API接口定义、请求参数、响应格式）
- ❌ 数据库表结构设计（表名、字段名、字段类型、索引、关联关系）
- ❌ 具体技术实现方案（前端框架、后端框架、中间件）
- ❌ 技术选型（编程语言、数据库、缓存、消息队列）
- ❌ 性能优化方案（SQL优化、缓存策略、并发处理）
- ❌ 安全设计方案（权限控制、数据加密、防攻击）
- ❌ 部署架构设计（服务器配置、容器化、CI/CD）

## 三、测试工程师规则

### 3.1 测试用例优先级

- **P0（阻塞级）**：核心功能，必须100%通过
- **P1（高优先级）**：重要功能，影响用户体验
- **P2（中优先级）**：一般功能，有替代方案
- **P3（低优先级）**：边缘场景

### 3.2 缺陷等级

- **致命（Critical）**：系统崩溃、数据丢失、安全漏洞
- **严重（Major）**：核心功能不可用
- **一般（Minor）**：非核心功能异常
- **轻微（Trivial）**：UI问题、文字错误

### 3.3 UI自动化框架推荐

- **Playwright**（推荐）：支持多浏览器、自动等待
- **Cypress**：适合前端团队，实时重载
- **Selenium**：生态成熟

## 九、补充规则

### 9.1 架构师 - 质量属性设计（量化指标）

**高性能**:
- 响应时间：P99 < 200ms（通过压测验证）
- 吞吐量：支持 10,000 QPS（JMeter测试）
- 并发用户：支持 50,000 同时在线（LoadRunner验证）

**高可用**:
- SLA：99.99%（年停机时间 < 52分钟）
- RTO：< 5分钟（故障恢复时间）
- RPO：< 1分钟（数据丢失量）

**可扩展**:
- 水平扩展：支持10倍流量增长（实测验证）
- 垂直扩展：支持单机资源翻倍
- 功能扩展：新功能开发周期 < 2周

**安全性**:
- 安全漏洞：高危漏洞0个，中危漏洞<3个
- 合规认证：通过等保三级/ISO27001
- 数据加密：传输加密+存储加密

**可维护性**:
- 监控覆盖率：核心指标100%监控
- 日志完整性：关键操作100%记录
- 故障定位时间：平均 < 15分钟

### 9.2 产品经理 - 字段设计规范

- 必须符合业务关联性校验
- 避免孤立字段
- 每个字段必须有明确的上下游关联（上游输入、下游输出）

### 9.3 产品经理 - React页面数据要求

- 开发的React页面必须同步维护50条业务数据
- 表格页面必须按照模板样例开发

### 9.4 测试工程师 - 协作模式

**与产品经理协作**:
- 需求评审阶段：识别测试点、评估可测性
- 测试计划阶段：确认测试范围、优先级
- 缺陷跟踪阶段：确认缺陷定级、修复优先级

**与开发工程师协作**:

**与运维工程师协作**:

### 12.3 JSR223语法规范

var demandMap = {};

### 13.1 什么是 JSX 低代码开发模式

JSX 低代码开发模式是一种基于 React 类组件的编程式开发方式：
- **编程式 UI**：通过编写 JSX 代码构建界面，而非配置 JSON
- **renderJSX 方法**：所有 UI 渲染逻辑集中在 `renderJSX()` 方法中
- **Antd 组件库**：强制使用 Ant Design 组件库
- **服务化架构**：通过 DataService 访问后端服务
- **类组件模式**：使用 React 类组件，不使用函数组件

### 13.2 重要提示 ⚠️

**assets.json 文件管理规则：**

### 13.6 数据服务使用

#### 查询条件说明
```javascript
{
  field: "字段名",
  conditionOperator: "操作符",
  conditionValues: [值]   // 即使单个值也要用数组
}
```

**常用操作符**：
| 操作符 | 说明 | 示例 |
|--------|------|------|
| `eq` | 等于 | `{ field: 'status', conditionOperator: 'eq', conditionValues: ['active'] }` |
| `like` | 模糊查询 | `{ field: 'name', conditionOperator: 'like', conditionValues: ['产品'] }` |
| `gt` | 大于 | `{ field: 'quantity', conditionOperator: 'gt', conditionValues: ['0'] }` |
| `eqa` | IN查询 | `{ field: 'status', conditionOperator: 'eqa', conditionValues: ['active', 'pending'] }` |
| `between` | 区间查询 | `{ field: 'createTime', conditionOperator: 'between', conditionValues: ['2024-01-01', '2024-12-31'] }` |

#### 标准查询模式
```javascript
async loadData() {
  await this.setStatePromise({ loading: true });
  try {
    const formId = await this.formService.getFormId('your_form_code');
    const res = await this.dataSourceMap.queryFormData.load({
      formId,
      conditionFilter: {
        conditionType: "and",
        conditions: [{ field: 'status', conditionOperator: 'eq', conditionValues: ['active'] }]
      },
      page: { current: this.state.pageNo, size: this.state.pageSize }
    });
    if (res.code === 200 && res?.result?.records?.length > 0) {
      await this.setStatePromise({ tableData: res.result.records, total: res.result.total });
    }
  } catch (error) {
    message.error('查询失败');
  } finally {
    await this.setStatePromise({ loading: false });
  }
}
```

**注意**：数据在 `result` 中，不是 `data`
```javascript
{ code: 200, result: { records: [...], total: 10, current: 1, size: 10 } }
```

### 14.3 ICE评分框架（详细版）

#### 评分维度
| 维度 | 说明 | 评分范围 | 权重 |
|------|------|----------|------|
| **I**mpact（影响） | 对业务目标的贡献程度 | 1-10分 | 40% |
| **C**onfidence（信心） | 对方案可行性的把握 | 1-10分 | 30% |
| **E**ase（难度） | 实施的难易程度 | 1-10分 | 30% |

#### 评分标准
- **Impact（影响）**：
  - 10分：解决核心业务痛点，影响全局
  - 7分：解决重要问题，影响多个模块
  - 4分：解决局部问题，影响单一模块
  - 1分：解决边缘问题，影响很小

- **Confidence（信心）**：
  - 10分：技术成熟，团队熟悉，有成功案例
  - 7分：技术可行，团队有一定经验
  - 4分：技术较新，需要学习和验证
  - 1分：技术风险高，不确定性大

- **Ease（难度）**：
  - 10分：开箱即用，1周内完成
  - 7分：需要配置，1个月内完成
  - 4分：需要定制开发，3个月内完成
  - 1分：需要大量改造，6个月以上

#### 计算公式
```
ICE总分 = (Impact × 0.4 + Confidence × 0.3 + Ease × 0.3)

或简化版：
ICE总分 = (Impact × Confidence × Ease) / 100
```

#### 决策标准
| 分数 | 建议 | 处理时效 | 说明 |
|------|------|----------|------|
| ≥7.0 | **优先处理** | 48小时内响应 | 高价值、高信心、易实现 |
| 5.0-7.0 | **正常处理** | 72小时内响应 | 需要进一步评估 |
| <5.0 | **延后处理** | 一周后评估 | 风险高或价值低 |

## 十六、测试工程师详细规则（补充）

### 16.1 禁止事项（绝对不能做）

#### 测试数据安全
- **禁止**使用生产环境的真实用户数据进行测试
- **禁止**将测试数据（尤其是包含敏感信息的数据）提交到公共代码仓库
- **禁止**在未加密的情况下传输或存储用户隐私数据
- **禁止**在测试报告中暴露用户的真实身份信息

#### 测试环境操作
- **禁止**在共享测试环境中执行可能破坏数据的破坏性测试（除非已备份）
- **禁止**未经授权访问生产环境进行测试
- **禁止**在测试环境中使用弱密码或默认密码

#### 缺陷管理
- **禁止**提交无法复现的缺陷（必须提供明确的复现步骤）
- **禁止**将多个问题合并到一个缺陷报告中
- **禁止**未经确认就关闭缺陷（必须经过验证）

### 16.2 必须遵循的原则

#### 测试独立性原则
- 每个测试用例应该独立执行，不依赖其他测试用例的执行结果
- 测试执行顺序不应该影响测试结果
- 测试数据准备和清理应该在每个测试用例中完成

#### 可重复性原则
- 测试用例必须能够在相同条件下重复执行并得到相同结果
- 记录完整的测试环境配置和测试数据
- 使用版本控制管理测试脚本和测试数据

#### 可追溯性原则
- 测试用例必须与需求关联，确保需求覆盖
- 缺陷必须与测试用例关联，便于追溯问题来源
- 测试报告必须包含完整的测试执行记录

#### 客观性原则
- 测试结果必须基于事实，不受个人主观判断影响
- 缺陷定级必须依据标准，不夸大也不缩小
- 质量评估必须基于数据，提供客观证据

### 16.3 个人工作规范

#### 时间管理
- **每日开始**：检查缺陷状态，跟进待验证的修复，调整当日测试计划
- **每日结束**：更新测试进度，记录当日工作，总结测试经验，提取测试模式
- **每周计划**：制定周测试计划，分配测试任务，识别测试模式
- **每周总结**：汇总测试结果，分析质量趋势，沉淀测试知识，更新测试模式库

#### 文档管理
- 测试用例必须存储在统一的测试管理工具中
- 测试报告必须按项目、按版本归档
- 缺陷报告必须包含完整的上下文信息
- 知识总结必须及时记录，便于团队共享
- 测试模式库必须定期维护和更新
- 按知识层次组织和管理测试文档（案例库、模式库、原则库）

#### 沟通协作
- **及时反馈**：发现问题立即记录并反馈，不拖延，快速迭代测试，及时调整测试策略
- **清晰表达**：描述问题要准确、简洁、完整，包括测试模式和经验总结
- **主动沟通**：与开发、产品保持密切沟通，了解需求变更，及时调整测试计划
- **团队协作**：分享测试经验和模式，帮助团队成员成长，共同建设测试知识体系

### 17.3 测试规范

#### 测试用例规范

**用例结构**：
```
用例编号：TC-模块-序号（如：TC-LOGIN-001）
用例标题：简明描述测试目的
前置条件：执行测试前必须满足的条件
测试步骤：
  1. 步骤1
  2. 步骤2
  ...
预期结果：明确可验证的预期结果
优先级：P0/P1/P2/P3
测试模式：关联的测试模式（如：登录验证模式、边界值测试模式等）
```

**测试模式库建设**：
- 建立测试模式库，存储和管理通用测试模式
- 从测试用例中提取通用测试模式
- 定期更新和维护测试模式库
- 基于测试模式自动生成测试用例

**优先级定义**：
- **P0（阻塞级）**：核心功能，必须100%通过
- **P1（高优先级）**：重要功能，影响用户体验
- **P2（中优先级）**：一般功能，有替代方案
- **P3（低优先级）**：边缘场景，可延期修复

#### 缺陷管理规范

**缺陷等级**：
- **致命（Critical）**：系统崩溃、数据丢失、安全漏洞
- **严重（Major）**：核心功能不可用、主要流程阻塞
- **一般（Minor）**：非核心功能异常、有替代方案
- **轻微（Trivial）**：UI问题、文字错误、不影响功能

**缺陷生命周期**：
```
新建 → 已确认 → 处理中 → 已修复 → 待验证 → 已关闭
  ↓       ↓         ↓         ↓         ↓
  └───────┴─────────┴─────────┴─────────┘
         可重新打开（验证不通过）
```

#### 测试报告规范

**报告内容**：
- **测试概况**：测试范围、测试时间、测试人员
- **执行统计**：用例总数、通过数、失败数、阻塞数
- **缺陷统计**：缺陷总数、按等级分布、按模块分布
- **质量评估**：产品质量评价、发布建议
- **风险说明**：已知问题、遗留风险
