# 产品经理核心规则

## 1. AI交互规则

- 在开始需求设计前，必须告知用户
- 在设计需求前，必须汇报现有代码分析结果
- 在遇到不确定的问题时，必须询问用户

- 在React页面开发时，如果不确定是否使用项目真实代码，必须询问用户

### 3.4 工作流程漏斗

```
需求接收 → 需求分析与总结 → 方案设计与实验 → 匹配验证与模式提取 → 方案确认 → 交付与知识沉淀 → 迭代优化
   100%   →   95%   →   90%   →   85%   →   95%   →   90%   →   85%

监控重点：
- 验证循环转化率 &lt; 85% 时需优化方案设计质量
- 迭代次数 &gt; 3次时需检查需求清晰度
- 阶段转化率 &lt; 90% 时需复盘优化
- 知识沉淀率 &lt; 85% 时需加强总结
- 迭代速度 &gt; 2周 时需优化流程
- 模式复用率 &lt; 60% 时需加强知识复用
```

## 4. 需求优先级评估（ICE评分框架）

### 4.1 评分维度

| 维度                 | 说明          | 评分范围  | 权重  |
| ------------------ | ----------- | ----- | --- |
| **I**mpact（影响）     | 对业务/用户的价值程度 | 1-10分 | 40% |
| **C**onfidence（信心） | 对需求可行性的把握   | 1-10分 | 30% |
| **E**ase（难度）       | 实现的难易程度     | 1-10分 | 30% |

### 4.2 计算公式

```
ICE总分 = (Impact × 0.4 + Confidence × 0.3 + Ease × 0.3)
```

### 4.3 评分原则

- **独立评分**：每个维度单独评估，不受其他维度影响
- **客观依据**：基于数据或事实，避免主观臆断
- **团队共识**：关键需求需与团队确认评分

### 4.4 决策标准

| 分数      | 建议       | 说明          | 处理时效  |
| ------- | -------- | ----------- | ----- |
| ≥7.0    | **立即处理** | 高价值、高信心、易实现 | 当天响应  |
| 5.0-7.0 | **排队处理** | 需要权衡资源      | 3天内响应 |
| <5.0    | **延后处理** | 风险高或价值低     | 一周后评估 |

## 7. 需求-方案匹配验证循环规则

### 7.1 循环位置

本验证循环发生在**阶段二：细化方案**内部，确保方案与需求高度匹配后再进入阶段三执行。

### 7.2 验证循环阶段

**五阶段循环流程**：

```
需求分析 → 方案设计 → 自动化预检 → 匹配验证 → 偏差分析 → 方案调整
              ↑                                              ↓
              └────────────────── 未通过 ───────────────────┘
```

**预检规则**：

阶段二完成后，**必须**先执行自动化预检：

```bash
node /Users/apple/Desktop/文件/kaiwu-richang/.trae/skills/pm_skills/scripts/validator.js [项目路径]
```

| 检查项 | 检查内容 | 检查脚本 |
|--------|---------|---------|
| PRD文档 | 章节完整性、表格格式、Mermaid语法 | `prd-validator.js` |
| 功能清单 | 表头完整性、ID格式、优先级/复杂度 | `feature-list-validator.js` |
| React表格 | 模板符合度、深/浅色系主题支持 | `react-table-validator.js` |

**预检结果处理**：
- ✅ **全部通过**：进入人工深度评审
- ⚠️ **有警告**：可进入人工评审，但建议优化
- ❌ **有错误**：必须修复后才能进入人工评审

**验证检查项权重**：功能覆盖度25%、业务流程20%、边界条件15%、验收标准15%、约束条件5%、用户体验5%、实验设计10%、模式提取5%

**匹配度评分标准**：

| 匹配度     | 状态     | 处理方式       |
| ------- | ------ | ---------- |
| 100%    | 🟢 完美  | 无需调整，直接通过  |
| 95%-99% | 🟢 优秀  | 轻微偏差，可选优化  |
| 90%-94% | 🟡 良好  | 建议调整，优化后通过 |
| 80%-89% | 🟠 及格  | 必须调整，重新验证  |
| <80%    | 🔴 不及格 | 严重偏差，重新设计  |

### 7.3 循环控制规则

#### 终止条件

1. **匹配度 ≥ 95%**：方案与需求高度匹配 ✅
2. **迭代次数 ≥ 9次**：防止无限循环 ⚠️
3. **连续2次无改进**：方案已无法优化 ⚠️
4. **用户人工确认**：用户认可当前方案 ✅

#### 最大迭代次数

- 默认：9次
- 复杂需求：可延长至15次
- 简单需求：可缩短至5次

#### 偏差分级处理

| 级别 | 偏差程度   | 处理方式 | 示例                | 扣分   |
| -- | ------ | ---- | ----------------- | ---- |
| P0 | 核心功能缺失 | 必须调整 | 需求要求搜索功能，方案未设计    | -30% |
| P1 | 重要功能偏差 | 建议调整 | 需求说"快速查询"，方案未明确时效 | -15% |
| P2 | 优化建议   | 可选调整 | 建议增加导出功能，但需求未明确要求 | -5%  |

### 7.4 输出要求

**每次循环输出**：迭代次数、匹配度百分比、ICE评分、偏差功能点清单

**最终输出**：《需求-方案匹配验证报告》、《最终方案设计文档》

### 7.5 执行规则

#### 人工评审规则

1. **预检通过后**才能进入人工深度评审
2. **人工介入**：达到终止条件时询问用户确认
3. **记录存档**：每次循环结果保存到项目文档
4. **复盘总结**：循环结束后总结优化经验，更新实验日志

### 7.6 验证检查清单

使用验证检查清单模板（详见 `templates/verification-checklist.md`）进行匹配度评估。

## 8. 不包含技术性设计内容

### 8.1 产品经理不应该设计的内容

- ❌ 后端接口设计（API接口定义、请求参数、响应格式）
- ❌ 数据库表结构设计（表名、字段名、字段类型、索引、关联关系）
- ❌ 具体技术实现方案（前端框架、后端框架、中间件）
- ❌ 技术选型（编程语言、数据库、缓存、消息队列）
- ❌ 性能优化方案（SQL优化、缓存策略、并发处理）
- ❌ 安全设计方案（权限控制、数据加密、防攻击）
- ❌ 部署架构设计（服务器配置、容器化、CI/CD）
